Grimmjow · O Problema
Análise de boletos era 100% manual
Identificar se um cliente tinha boletos a mais, a menos ou nenhum exigia cruzamento manual de planilhas sem nenhuma padronização.
📋

Processo inteiramente em Excel

O analista baixava a planilha de recebíveis e a base de clientes e precisava criar fórmulas manualmente para cada instalação.

🔢

Cálculo manual de boletos esperados

Era necessário calcular =HOJE() − Data Ativo, dividir por 30 e comparar com a contagem real — para cada cliente individualmente.

❓

Meses faltantes não identificados

Saber quais meses específicos estavam ausentes exigia análise manual linha a linha — sem nenhuma automação de listagem.

🚫

Casos especiais ignorados

Clientes sem Data Ativo e clientes dentro do período de carência misturavam-se com os demais, gerando falsos negativos.

⏱

Custo operacional real

Processar uma única fornecedora com milhares de instalações levava horas. Com 7 fornecedoras ativas, o analista gastava dias inteiros apenas identificando inconsistências que o sistema agora encontra em segundos.

3 plataformas + 2 abas no BKO

ExperIP para ligação, BKO para atendimento e cancelamento em abas separadas, Chatwoot para HSM — tudo manual e sem integração.

Risco jurídico: HSM manual

O HSM de cancelamento é a proteção contratual da iGreen. Hoje depende exclusivamente do colaborador enviá-lo — 482 casos com B.O/Procon/R.A em 2 meses.

2.166 clientes saíram da base sem que a iGreen soubesse o motivo — impossibilitando qualquer ação de retenção ou melhoria de produto.

🔄

3.537 clientes com status incorreto

Clientes registrados como cancelados que deveriam ser reprovados — comprometendo a confiabilidade do churn reportado.

Integração TI · Grimmjow
Como integrar o Grimmjow ao sistema interno
Caminhos para o time de TI evoluir o sistema desktop atual para uma solução nativa no BackOffice iGreen.
Fase 1 · Imediato

Deploy do app desktop na rede

  • Hospedar o Grimmjow.py em servidor interno
  • Criar atalho acessível a todos os analistas via rede
  • Nenhuma dependência de configuração local
  • Stack: Python 3 + Tkinter + Pandas já instalados
Fase 2 · API BackOffice

Consumir dados direto da API

  • Substituir upload de planilhas por endpoints do BackOffice
  • GET /clientes → base de clientes com Data Ativo
  • GET /recebiveis/:fornecedora → boletos por instalação
  • Elimina etapa de download manual das bases
Fase 3 · Web no BKO

Interface web dentro do BackOffice

  • Migrar a lógica Python para microserviço Flask/FastAPI
  • Criar rota no BKO: /analise-faturamento
  • Analista filtra fornecedora e dispara pelo próprio BKO
  • Resultado renderizado em tela — sem download de arquivo
Fase 4 · Automação

Análise agendada e chamados automáticos

  • Cronjob semanal dispara a análise automaticamente
  • NEGATIVOS → abre chamado no ClickUp via API
  • SEM DATA → notifica Gestão de Dados automaticamente
  • Histórico de execuções com comparativo entre semanas
🔧

Stack tecnológico atual do Grimmjow

Python 3 · Pandas · OpenPyXL · Tkinter. A lógica de negócio está 100% isolada na função processar() — pode ser importada diretamente por qualquer serviço Flask/FastAPI sem reescrita.
